# Overview

## What This Is

`magic_storage` — NeoForge 1.21.1 的一站式儲存 + 合成 mod。玩家用一個終端完成儲存、取用與合成：每種物品數量無上限，種類容量由有限 Storage Units 提供，或由 Creative Storage Unit 明確切換為無限；另有配方書式一鍵合成，以及由 Fuel 頁已安裝機器產生、合成時一次消費的「合成能量」。

概念源自 **Terraria 的 Magic Storage**(無限容量、配方書操作);NeoForge 的儲存網路 / grid UI / 物品處理等**技術模式參考 Refined Storage 2** 原始碼(見 `docs/notes.md`)。

## Key Concepts / Domain

- **Storage Core** — 網路中樞，所有計算（容量、合成、能量）都在 Core；Unit 只提供空間。永久資料從建立時起由 overworld `data/magic_storage_core_storages.dat` 的 `CoreStorageRepository` 擁有，Core 所在 chunk 只保存 `storageId` 與 `storageSchema`。庫存以每段最多 63 種的任意多段格式持久化，63 不是玩家容量上限。破壞、爆炸、創造模式或 Wrench 拆除含資料 Core 時，掉落物只攜帶一次性 recovery UUID 與摘要，放置時把同一 storage/network record 移到新 Core，不複製 payload；`/magic_storage recover_core` 只重發同一個未 claim capability。缺少、毀損、packed 或重複 attachment 一律 fail closed 並拒絕終端/自動化操作，不會製造空白 replacement。本次早期開發格式不遷移舊 inline Core NBT 或舊 recovery snapshot。
- **Storage Unit** — 六個有限 Tier 的儲存方塊獨立、不可逆、可並聯不限數量；T1→T6 以 copper cell、iron brace、gold/lapis lattice、diamond/quartz cross、prismarine/ender halo、netherite/amethyst crown 逐級增加 ornament complexity。`creative_storage_unit` 只由創造物品欄/指令取得、沒有配方/庫存/生成能力，en_us/zh_tw item tooltip 也明確標示無限種類且不生成物品；有效且目前已載入的單 Core 網路只要連上一個就以 explicit unlimited 種類容量運作，多個效果相同。移除或卸載只回到有限超額規則，不刪既有物品。
- **配方進程 / 指南** — T1 改以黑色混凝土粉末、銅、紅石與木桶製作，不再要求紫水晶；Core 與 T1–T6 依序引入黑曜石、玄武岩、鑽石/史萊姆、海磷晶/蜂蜜與獄髓。Patchouli 的雙語「配方總覽」集中顯示所有 Magic Storage 配方，每個配方只出現一次。
- **工具與終端進程** — Wrench 是 Brush + Tripwire Hook 的廉價建設工具。Storage Terminal 保持起步期但加入紅石火把控制件；Crafting Terminal 以金、銅、紅石和工作台成為早中期升級；Remote Terminal 再要求終界珍珠、黑曜石與指南針，作為較晚取得的移動能力。指南書本身註冊為 `magic_storage:guide_book` 並可合成。
- **Crafting Energy / Fuel categories** — 多個能量池統一用 `Map<EnergyType, Long>`。NeoForge `magic_storage:machine_descriptor` 公開 registry 由 server 擁有順序與狀態；內建九個 Fuel-page descriptors 分成三類：Furnace/Blast Furnace/Smoker/Campfire/Brewing Stand 是可堆疊 process machines，stack `N` 為對應 pool 產生 `N/tick`；Crafting Table/Stonecutter/Smithing Table 是最多各一個、可取回且只解鎖配方的 instant stations；axe 是不可安裝/取回的 consumable input。Fuel UI 先放獨立compact target bar，再以 Consumables / Timed Stations / Instant Stations 三個全寬category panels呈現；三者從標題下方填滿到Inventory label band，每個右側descriptor區依可用寬高形成multi-row pages並獨立滾輪翻頁。Core 使用固定 256 格 bank，依 stable descriptor ID 持久化 machine/station stacks 與 finite/explicit-infinite consumable state；descriptor缺少、category/ingredient不相容、超過上限或addon item暫時不存在時，原始machine/descriptor entry NBT會留在repository record並於後續load重試，不會在這次格式切換中搬移或刪除。registry順序只凍結/排序一次，Core tick重用同一immutable snapshot。有限值為 remaining durability × (vanilla Unbreaking level + 1)，Unbreakable 轉為永不遞減的無限值。`ItemStack#getBurnTime(null)` 是 Fuel 真值，concrete cooking time 同時決定 process/Fuel 消耗。所有 mutation 在 server，client 只接同步。
- **Crafting/Fuel GUI** — `TerminalProfile` 以 capability 組合 Storage reduced profile 與 Crafting full profile，`TerminalLayout.forProfile(...)` 是唯一 geometry entry point；同一 immutable geometry 驅動 rendering、widgets、slot reconstruction、hitboxes、tooltips、scroll與 EMI exclusions。Storage/Craftable 共用 server-owned exact amount、Name/Quantity/Mod/ID comparator與固定 slot-local scale；recipe workspace只消費 server-synced exact `RecipePresentation`，EMI存在時只在 diagram使用公開 widgets，ledger/footer始終由Magic Storage擁有。Cycle control統一 left-next/right-previous/wheel-down-next/wheel-up-previous/middle-reset，tooltip只列名稱與目前值；只有boolean on/off control畫彩色狀態燈，Craft Output/Fuel Target/sort/search等value selectors不亮燈。Fuel/recipe semantic workspace與player inventory共用vanilla light container grammar：raised panels、inset wells、slot frames、vanilla widgets與dark-gray text；world block palette不得套入GUI。Recipe diagram+ledger是footer上方的compact card；ledger最多四欄、由頂端對齊，目前可容納三列，超過八種required resources會進第三列；`x1/x8/x64/Max`與navigation使用vanilla-style buttons。Fuel target仍有bounded foreground popup；三個Fuel descriptor grids各自使用adaptive columns × rows、page indicator與exact hover，並以每類64 descriptors回歸測試大量整合。每格為上方slot/icon、下方置中amount，空station顯示dim代表物。完整localized type-capacity訊息位於player inventory正右方的獨立info panel；無限狀態使用 Creative Storage Unit 圖示，有限狀態維持 T1 圖示，Instant Stations不再為它保留格位。第三方 descriptor 依 ResourceLocation 排序，menu opening payload 攜帶 server frozen snapshot，通用 consumable finite/∞ state 另由 clientbound packet 同步；詳見 [`machine-descriptor-api.md`](machine-descriptor-api.md)。
- **Terminal(三階)** — `storage_terminal`(T1 取存)、`crafting_terminal`(T2 配方+一鍵合成)、`remote_terminal`(T3 物品,同維度遠端存取；綁 exact Core UUID,同座標 replacement 不會被靜默接管；重綁不覆蓋其他 `CUSTOM_DATA`)。
- **Network** — Core 掃描/維護 Core ↔ Unit ↔ Terminal/Bus 連通;一般/指令/活塞放置會合併到 next-tick topology pass，安全範圍內增量成長，破壞/不確定拓樸時 full rebuild。Server 也會依 FULL chunk lifecycle 延後、合併並 bounded 掃描有效的 loaded network boundary：遠端 chunk 卸載時仍 loaded 的 Core 自動失去該段有限/Creative 容量，重新載入後自動恢復；server `LevelChunk` load event 只 one-shot 處理，`getChunkNow` 未就緒不逐 tick retry，後續 FULL ticket transition 再補排；短暫 ticket hop 會取消成對變更，不 force-load chunk、不每 tick 掃整網，multi-Core conflict 規則不變。Terminal/Bus 只沿目前已載入且仍為 network block 的 cached path 工作，失效時改找另一條已載入路徑，絕不為驗證連線載入 chunk。Import Bus 正面仍主動從容器抽取，同時在所有面暴露 stable、insert-only 的 NeoForge item-handler capability，讓 hopper/mod automation 被動推入並走 Core simulate→commit；active/passive missing-Core lookup 共用10-tick negative cache，避免每次 automation probe 重跑 bounded BFS。Directional active Import、passive Import與active Export都會先套用server-owned automation gate與deterministic filter policy；mode/side/ownership、capability invalidation與directionless仍待後續。Export Bus 目前只保留朝正面的主動輸出，無方向模式列入未來工作。
- **Supported recipes / safe output** — Vanilla exact-concrete Shaped/Shapeless Crafting、Stonecutting、Smelting、Blasting、Smoking、Campfire、component-exact `SmithingTransformRecipe`、component-preserving `SmithingTrimRecipe`，以及只掃 `minecraft` namespace default state 的 internal synthetic axe strip/scrape/wax-off；任何 namespace 使用相同 exact vanilla class/type 都零設定支援。Custom exact class/type 可透過公開 `magic_storage:recipe_family` registry 的 bounded `singleItemToItem` factory 一次註冊整個 family，並綁定既有 machine descriptor；沒有 reflection、名稱、serializer、`Recipe#getIngredients()` 或 EMI widget 推測 fallback。每種都先要求目前 Core 安裝 matching station，或具備足夠 Axe Energy。Smithing 三個 role 直接用原 recipe predicate 比對實際完整 stack，不用 registry-default Ingredient 近似重建。Transform/Trim從目前Core與可選玩家背包的exact sources解出variant；Trim用vanilla registry context解析pattern/material，同recipe ID的每個exact output identity分開列出。Variant來源以item ID + registry-aware canonical component NBT決定順序，組合超過65,536或缺少mapping時整體fail closed。所有lookup/select/current/reload/execute都按ID重取目前server holder，再以所選exact output identity重解variant；只有Transform的generic target lookup保留既有target-result discovery，實際執行仍綁定source-backed exact variant。Craftable display amount只取該output目前Core庫存；synthetic marker在建立`ItemKey`或實際輸出前移除。Direct `×1/×8/×64/Max` 使用獨立、server-synced 的 Player/Storage session destination：Player 先放玩家36格、overflow回Core；Storage則讓primary output與crafting remainder全部只進Core。所選策略無法接收完整批次時在mutation前整批拒絕，不掉落、不消失。Client必要EMI接受`[1.1.24,2)`並傳exact recipe ID + amount + destination；NONE只選取，CURSOR/INVENTORY執行單層server-authoritative magic craft，且不受direct-terminal toggle影響；dedicated server不要求EMI。Max與commit沿用long-safe capacity、deferred-listener mutation batch與完整rollback；finite Axe Energy每craft精確扣一，infinite不遞減。目前仍不執行遞迴合成；更多 bounded factories、第三方 family compatibility 與多步驟路線依 [`recipe-family-api.md`](recipe-family-api.md) 及 `docs/superpowers/plans/2026-07-14-modded-recipes-multistep-crafting.md` 擴充，EMI tree只作client規劃/參考介面。Special/dynamic/incomplete subclasses、缺少trim registry mapping、Brewing、任意mod block hook與自由world/player-context recipe仍fail closed。
- **Typed-resource Core** — GitHub #9 的第一階段已加入 stable kind/resource ID + exact variant payload + long amount 的 server-owned non-item ledger、raw-preserving record persistence與atomic mixed-key algebra。Core目前暴露exact-component NeoForge fluid、與Fuel頁完全分離的NeoForge Energy，以及只在Mekanism存在時載入的long-amount chemical capability；Mekanism缺席時base server不link其API。Item、fluid、power、chemical是第一批而非封閉enum，unknown provider key仍原樣持久化。Existing item仍在legacy exact-`ItemKey` map，terminal/bus/creative/public addon kind API與live item+non-item crafting transaction尚未完成，不能把ledger unit contract誤寫成玩家已可混合合成。未來recipe family才在同一transaction boundary宣告deterministic N-input/N-output；external-machine send-and-wait不屬於Magic Storage模型。進度與契約見 [`typed-resource-storage.md`](typed-resource-storage.md)。
- **Assets / tools** — Runtime block/item icons維持vanilla 16x16與同一dark chassis/cyan-amethyst palette。T1–T6 ornament逐級華麗，Creative Storage Unit使用Replicate seed 71425候選與同family semantic cleanup完成cyan-amethyst infinity cell；全部以x=7.5水平鏡射並以2×2偶數中心校正。World casing預設使用ordinary 16x16 vanilla model；若client安裝可選Fusion 1.2.12，內建且自動啟用的connected-casing overlay才會以`pieced` 80×16 sheet相連，Creative Unit也納入全部12種network block predicates。item model仍是ordinary 16x16；Fusion只進Gradle隔離的client/data `fusionRuntime` source set，不進server/GameTest dependency也不bundle。Magic Storage Wrench加入`c:tools/wrench`並接受同tag的其他mod工具：右鍵旋轉directional bus，Shift+右鍵以正常loot path安全拆除network block；含資料Core透過同一repository record的recovery token移動。生成metadata/contact sheets只留在`art/texture-generation/`，static tests不取代fullscreen目視。
- **macOS GUI fullscreen validation** — Visual Prism sessions use Minecraft F11 through a macOS-only borderless Cocoa implementation, not GLFW monitor fullscreen: it never changes the desktop display mode. The deterministic lab places Creative Storage Unit in the active network and both isolated/connected galleries, and supplies its item so the checklist can inspect localized ∞ capacity, tooltip, native model, and Fusion seams. The runner records the desktop mode before launch and rejects handoff if it differs at READY. User visual approval remains mandatory; normal close is F11 → visible titled window → Command-Q. See [`docs/macos-fullscreen-guide.md`](macos-fullscreen-guide.md) and [`docs/notes.md`](notes.md#prism-dev--manual-handoff).
