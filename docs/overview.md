# Overview

## What This Is

`magic_storage` — NeoForge 1.21.1 的一站式儲存 + 合成 mod。玩家用一個終端完成儲存、取用與合成:無限容量(只限物品種類數)、配方書式一鍵合成、以及由 Fuel 頁已安裝機器產生、合成時一次消費的「合成能量」。

概念源自 **Terraria 的 Magic Storage**(無限容量、配方書操作);NeoForge 的儲存網路 / grid UI / 物品處理等**技術模式參考 Refined Storage 2** 原始碼(見 `docs/notes.md`)。

## Key Concepts / Domain

- **Storage Core** — 網路中樞，所有計算（容量、合成、能量）都在 Core；Unit 只提供空間。永久資料從建立時起由 overworld `data/magic_storage_core_storages.dat` 的 `CoreStorageRepository` 擁有，Core 所在 chunk 只保存 `storageId` 與 `storageSchema`。庫存以每段最多 63 種的任意多段格式持久化，63 不是玩家容量上限。破壞、爆炸、創造模式或 Wrench 拆除含資料 Core 時，掉落物只攜帶一次性 recovery UUID 與摘要，放置時把同一 storage/network record 移到新 Core，不複製 payload；`/magic_storage recover_core` 只重發同一個未 claim capability。缺少、毀損、packed 或重複 attachment 一律 fail closed 並拒絕終端/自動化操作，不會製造空白 replacement。本次早期開發格式不遷移舊 inline Core NBT 或舊 recovery snapshot。
- **Storage Unit** — 六個 Tier 的儲存方塊，獨立、不可逆、可並聯不限數量；T1→T6 不再只用容量條區分，而以 copper cell、iron brace、gold/lapis lattice、diamond/quartz cross、prismarine/ender halo、netherite/amethyst crown 逐級增加 ornament complexity。
- **配方進程 / 指南** — T1 改以黑色混凝土粉末、銅、紅石與木桶製作，不再要求紫水晶；Core 與 T1–T6 依序引入黑曜石、玄武岩、鑽石/史萊姆、海磷晶/蜂蜜與獄髓。Patchouli 的雙語「配方總覽」集中顯示所有 Magic Storage 配方，每個配方只出現一次。
- **工具與終端進程** — Wrench 是 Brush + Tripwire Hook 的廉價建設工具。Storage Terminal 保持起步期但加入紅石火把控制件；Crafting Terminal 以金、銅、紅石和工作台成為早中期升級；Remote Terminal 再要求終界珍珠、黑曜石與指南針，作為較晚取得的移動能力。指南書本身註冊為 `magic_storage:guide_book` 並可合成。
- **Crafting Energy / Fuel categories** — 多個能量池統一用 `Map<EnergyType, Long>`。NeoForge `magic_storage:machine_descriptor` 公開 registry 由 server 擁有順序與狀態；內建九個 Fuel-page descriptors 分成三類：Furnace/Blast Furnace/Smoker/Campfire/Brewing Stand 是可堆疊 process machines，stack `N` 為對應 pool 產生 `N/tick`；Crafting Table/Stonecutter/Smithing Table 是最多各一個、可取回且只解鎖配方的 instant stations；axe 是不可安裝/取回的 consumable input。Fuel UI 先放獨立compact target bar，再以 Consumables / Timed Stations / Instant Stations 三個全寬category panels呈現；三者從標題下方填滿到Inventory label band，每個右側descriptor區依可用寬高形成multi-row pages並獨立滾輪翻頁。Core 使用固定 256 格 bank，依 stable descriptor ID 持久化 machine/station stacks 與 finite/explicit-infinite consumable state；descriptor缺少、category/ingredient不相容、超過上限或addon item暫時不存在時，原始machine/descriptor entry NBT會留在repository record並於後續load重試，不會在這次格式切換中搬移或刪除。registry順序只凍結/排序一次，Core tick重用同一immutable snapshot。有限值為 remaining durability × (vanilla Unbreaking level + 1)，Unbreakable 轉為永不遞減的無限值。`ItemStack#getBurnTime(null)` 是 Fuel 真值，concrete cooking time 同時決定 process/Fuel 消耗。所有 mutation 在 server，client 只接同步。
- **Crafting/Fuel GUI** — `TerminalProfile` 以 capability 組合 Storage reduced profile 與 Crafting full profile，`TerminalLayout.forProfile(...)` 是唯一 geometry entry point；同一 immutable geometry 驅動 rendering、widgets、slot reconstruction、hitboxes、tooltips、scroll與 EMI exclusions。Storage/Craftable 共用 server-owned exact amount、Name/Quantity/Mod/ID comparator與固定 slot-local scale；recipe workspace只消費 server-synced exact `RecipePresentation`，EMI存在時只在 diagram使用公開 widgets，ledger/footer始終由Magic Storage擁有。Cycle control統一 left-next/right-previous/wheel-down-next/wheel-up-previous/middle-reset，tooltip只列名稱與目前值；只有boolean on/off control畫彩色狀態燈，Craft Output/Fuel Target/sort/search等value selectors不亮燈。Fuel/recipe semantic workspace與player inventory共用vanilla light container grammar：raised panels、inset wells、slot frames、vanilla widgets與dark-gray text；world block palette不得套入GUI。Recipe diagram+ledger是footer上方的compact card；ledger最多四欄、由頂端對齊，目前可容納三列，超過八種required resources會進第三列；`x1/x8/x64/Max`與navigation使用vanilla-style buttons。Fuel target仍有bounded foreground popup；三個Fuel descriptor grids各自使用adaptive columns × rows、page indicator與exact hover，並以每類64 descriptors回歸測試大量整合。每格為上方slot/icon、下方置中amount，空station顯示dim代表物。完整localized type-capacity訊息位於player inventory正右方的獨立info panel，Instant Stations不再為它保留格位。第三方 descriptor 依 ResourceLocation 排序，menu opening payload 攜帶 server frozen snapshot，通用 consumable finite/∞ state 另由 clientbound packet 同步；詳見 [`machine-descriptor-api.md`](machine-descriptor-api.md)。
- **Terminal(三階)** — `storage_terminal`(T1 取存)、`crafting_terminal`(T2 配方+一鍵合成)、`remote_terminal`(T3 物品,同維度遠端存取；綁 exact Core UUID,同座標 replacement 不會被靜默接管；重綁不覆蓋其他 `CUSTOM_DATA`)。
- **Network** — Core 掃描/維護 Core ↔ Unit ↔ Terminal/Bus 連通;一般/指令/活塞放置會合併到 next-tick topology pass，安全範圍內增量成長，破壞/不確定拓樸時 full rebuild。Terminal/Bus 只沿目前已載入且仍為 network block 的 cached path 工作，失效時改找另一條已載入路徑，絕不為驗證連線載入 chunk。Import Bus 正面仍主動從容器抽取，同時在所有面暴露 stable、insert-only 的 NeoForge item-handler capability，讓 hopper/mod automation 被動推入並走 Core simulate→commit；active/passive missing-Core lookup 共用10-tick negative cache，避免每次 automation probe 重跑 bounded BFS。Export Bus 目前只保留朝正面的主動輸出，無方向模式列入未來工作。
- **Supported recipes / safe output** — Vanilla exact-concrete Shaped/Shapeless Crafting、Stonecutting、Smelting、Blasting、Smoking、Campfire、component-exact `SmithingTransformRecipe`，以及只掃 `minecraft` namespace default state 的 internal synthetic axe strip/scrape/wax-off；每種都先要求目前 Core 安裝 matching station，或具備足夠 Axe Energy。Smithing 三個 role 直接用原 recipe predicate 比對實際完整 stack，不用 registry-default Ingredient 近似重建。Preview/execute 從目前 server `RecipeManager` 或依 Core Axe Energy 開啟的 exact synthetic axe catalog 重取 holder。Craftable 由 server recipe catalog 產生目前可合成的 output union，display amount 只取該 output 的目前 Core 庫存；synthetic marker 在建立 `ItemKey` 或實際輸出前會移除。Direct `×1/×8/×64/Max` 使用獨立、server-synced 的 Player/Storage session destination：Player 先放玩家 36 格、overflow 回 Core；Storage 則讓 primary output 與 crafting remainder 全部只進 Core。所選策略無法接收完整批次時在 mutation 前整批拒絕，不掉落、不消失。Client必要EMI接受`[1.1.24,2)`並傳 exact recipe ID + amount + destination；NONE只選取，CURSOR/INVENTORY執行單層server-authoritative magic craft，且不受direct-terminal toggle影響；dedicated server不要求EMI。Max 先計算資源上限，再找出 destination 可完整接收的最大 craft 數；midpoint 與每個 Core key 的 reservation/extraction/output/rollback 全程使用 long-count mutation，連 `Long.MAX_VALUE` 都不會退化成逐 `Integer.MAX_VALUE` chunk loop。Commit 在 deferred-listener mutation batch 內驗證、扣料、插入 output、扣 process/Fuel/Axe Energy，再套用玩家 inventory/cursor；finite Axe Energy 每 craft 精確扣一，infinite 不遞減。目前仍不執行遞迴合成；custom mod recipes與多步驟路線將依`docs/superpowers/plans/2026-07-14-modded-recipes-multistep-crafting.md`先建立server adapter/descriptor graph，EMI tree只作client規劃/參考介面，不作權威。Smithing Trim、special/dynamic subclasses、Brewing、任意 mod block hook 與自由 world/player-context recipe目前仍fail closed。
- **Assets / tools** — Runtime block/item icons維持vanilla 16x16與同一dark chassis/cyan-amethyst palette。T1–T6 ornament逐級華麗、以x=7.5水平鏡射並以2×2偶數中心校正，不再依賴容量條；bus保留cyan-in/orange-out方向front。World casing預設使用ordinary 16x16 vanilla model；若client安裝可選Fusion 1.2.12，內建且自動啟用的connected-casing overlay才會以`pieced` texture相連。item model仍是ordinary 16x16；Fusion只進Gradle隔離的client/data `fusionRuntime` source set，不進server/GameTest dependency也不bundle。Magic Storage Wrench加入`c:tools/wrench`並接受同tag的其他mod工具：右鍵旋轉directional bus，Shift+右鍵以正常loot path安全拆除network block；含資料Core透過同一repository record的recovery token移動。生成metadata/contact sheets只留在`art/texture-generation/`，static tests不取代fullscreen目視。
- **macOS GUI fullscreen validation** — Visual Prism sessions use Minecraft F11 through a macOS-only borderless Cocoa implementation, not GLFW monitor fullscreen: it never changes the desktop display mode. The runner records the desktop mode before launch and rejects handoff if it differs at READY. User visual approval remains mandatory; normal close is F11 → visible titled window → Command-Q. See [`docs/macos-fullscreen-guide.md`](macos-fullscreen-guide.md) and [`docs/notes.md`](notes.md#prism-dev--manual-handoff).
