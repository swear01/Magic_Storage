2026-07-17 Bus filter-policy gotcha：Allow空規則=allow none，Deny空規則=allow all；duplicate只讓第一條ACTIVE，後續保留為DUPLICATE，缺席registry/raw rule保留為UNAVAILABLE且兩者都match nothing。Allow candidates先依rule slot，再依item registry ID/canonical components；Deny candidates直接依全域相同item順序。Canonical components必須用registry-aware `DataComponentMap.CODEC`後遞迴排序CompoundTag keys並length-prefix，不可用`DataComponentMap.toString()`或輸入HashMap順序。Unsupported config即使呈現Deny-empty也必須整體fail closed。現行active Import維持來源slot順序後套filter，passive Import的simulate/execute共用同一policy且不修改caller stack，active Export依ordered candidates取第一個可搬運項目；`automationEnabled=false`三條路徑都fail closed，runtime config load會清transfer cooldown使新policy立即生效。方向模式、side mask、unsided與owner仍不可假裝已完整接線。

2026-07-17 Smithing variant gotcha：exact concrete `SmithingTransformRecipe`與`SmithingTrimRecipe`都必須從目前Core/可選玩家背包的完整component stacks解出source-bound contracts，不能讓generic `resolveVariants`用「空inputs的presentation是否非空」決定支援。每個bound contract的template/base/addition predicate同時驗原recipe predicate與exact `ItemKey`，輸出再呼叫目前recipe的`assemble`核對identity/count。Trim同一recipe ID可產生多個exact outputs；lookup/select/current/reload/execute必須以selected output identity重解current holder variant，commit不可退回只按ID的base match；若呼叫只提供ID且目前有多個variant，必須fail closed，不能猜canonical第一個。來源先按item registry ID + registry-aware canonical component NBT排序，不能依Core/hash/source iteration；template×base×addition超過65,536、component codec失敗、trim material/pattern mapping缺失或assemble為空時整體fail closed，不可回傳partial variants。Transform只有generic result lookup可顯式用recipe target identity找到canonical source-backed variant以保留既有操作；一旦已有exact selected output，所有後續路徑仍只能exact match。Special/dynamic/incomplete subclasses保持neutral unsupported。

### Prism dev / manual handoff

預設流程不再由 agent 用 Computer Use 逐步操作 Minecraft GUI；runner 只負責準備、離線啟動、等 READY 與收集本次 log，之後直接請使用者接手。這避免 LWJGL click/文字輸入與 Accessibility/Screen Recording 權限造成長時間等待。runner 用 `open -a`，不得另開 Terminal.app。若 agent 為診斷而開了額外 Terminal 視窗，流程結束前只關閉自己建立的視窗，不可動使用者原有 Terminal；結束時也要確認沒有自己留下的 Prism/Minecraft/Gradle process。使用者仍在手動目視時不可清掉 active client；只有使用者表示結束/卡死，或 log 已記 `Stopping!` 但 Java 仍殘留時，才先 TERM、逾時再 KILL。

`-o MagicStorageBot` 的 offline 意義是 Prism 不刷新 Microsoft/Xbox account，也不進登入流程。Vanilla authlib 仍可能對離線 UUID嘗試抓 skin/profile properties，並在 `latest.log` 留下 `Failed to fetch user properties` / 401 stack；這不是帳號登入或 token refresh。因 Prism 把該行視為 ERROR，`ShowConsoleOnError=true` 會讓 console 自動跳出，看起來像登入 log；runner 現在強制關閉這個 pop-up，但仍直接掃 `latest.log`。它只針對這一段完整已知 stack 白名單，其他 current-run `ERROR` / `FATAL` / `Caused by` 仍 fail，不能把一般錯誤一起藏掉。Prism console 設定語意見官方 [Launcher Settings](https://prismlauncher.org/wiki/help-pages/launcher-settings/)。

若 current-run 在 NeoForge early display 出現 `Failed to find a primary monitor` / `glfwGetPrimaryMonitor failed`，代表當下 macOS 沒有向 GLFW 暴露 active monitor，常見於螢幕睡眠或鎖定；`caffeinate -u` 在仍鎖定時也不保證可解。runner 會用明確的「wake and unlock」錯誤 fail closed 並清理本次 processes，不可把這段 `ERROR` 白名單化。使用者先實際喚醒並解鎖顯示器，再重跑同一 scenario。

- **Crafting Terminal Fuel categories 契約**:`MachineEnergyTable` 的九個 descriptors 有明確 category。Furnace/Blast Furnace/Smoker/Campfire/Brewing Stand 是可堆疊 process machines，stack `N` 每 tick 只為對應 pool 加 - **Terminal cycle/status/reset 契約**:所有 cyclic value selector 都是 left-next/right-previous/wheel-down-next/wheel-up-previous/middle-reset；reset defaults 為 Name、Ascending、normal search、Auto Fuel Target、Player Craft Output及既有 player-source default，且必須送 explicit server button action。只有 true boolean on/off control可畫彩色狀態燈；Craft Output、sort/search/Fuel Target等 value selectors不可亮綠/藍 marker。Page tabs可用 neutral selected treatment，但不能冒充 on/off light。Tooltip只顯示控制名稱與目前值，操作手勢寫在指南書。

- **終端顯示量不能塞進 `ItemStack#getCount`**:`StorageCoreBlockEntity.getDisplayStacks` 與 Craftable synthetic outputs 一律用 `TerminalDisplayStack` 在 `CUSTOM_DATA` 的 reserved marker 同步 exact non-negative `long` amount，實際 stack count 固定為 1 以保留零庫存 icon。`TerminalDisplayStack.amount` 是排序/畫字/tooltip 唯一來源；零不畫 overlay，tooltip 仍顯示 exact stored amount。`ItemKey.of`、recipe selection、EMI inventory view 與實際 storage/output 邊界必須 strip marker；EMI view 另外把 logical amount 傳給 `EmiStack`，不可把 marker 當成 recipe component或把庫存誤報為 1。Craftable amount 是目前 Core 庫存，不是 potential craft yield；零庫存可選/可合成但不可抽取。Name/Quantity/Mod/ID 都走同一 `TerminalEntryComparator`，component variants 必須有 deterministic final tie-breaker。

- **捲動封包是 absolute + server validated**:`TerminalScrollPacket` 傳 `{containerId,offset}`；server 只接受目前玩家開啟、container id 相同的 `StorageTerminalMenu`，再 clamp 到可見範圍。client drag 只在 absolute target 改變時送一包，不可用 button loop、固定 sleep 或強制 row-snap。screen re-init 必須保留搜尋文字、焦點與 debounce deadline。
- **Patchouli 1.20+ mod book 路徑**:`book.json` 留在 `data/magic_storage/patchouli_books/guide/book.json`,但 categories/entries/templates 必須放 `assets/magic_storage/patchouli_books/guide/en_us/...`;放在 `data/.../en_us` 會導致 `BookContentResourceListenerLoader preloaded 0 jsons` 與書內 `No Entries`。

- **Storage API / live menu contract**:一般 stack mutation API 是 `insertItem(stack, Action, Actor)` / `extractItem(key, amount, Action, Actor)` / `extractMatching(predicate, amount, Action, Actor)`；合成的大額 per-key transaction 改用 `insertItemCount(key, long, Action, Actor)` / `extractItemCount(key, long, Action, Actor)`，不得用數十億次 int chunk loop。種類容量的authoritative值是`StorageTypeCapacity(finiteTypeSlots, unlimited)`；`getTotalTypeSlots()`只保留有限component的legacy查詢，任何新type admission與craft output planning都必須使用explicit predicate。容量由topology導出，不另存NBT；full rebuild與incremental add共用composition，容量record變化會提升topology revision。`SIMULATE`不改storage/不fire event，`EXECUTE`才setChanged + 一次`onChanged`。Machine tick/consume、Fuel conversion、Axe Energy conversion/consume另fire`onEnergyChanged`，open menu在`broadcastChanges`刷新grid/preview。一次合成commit必須包在`beginMutationBatch`/`endMutationBatch`，把listener callback延到驗證、扣料、扣process/Fuel/Axe Energy與rollback邊界之後；否則reentrant refresh/conflict可能在兩個ingredient extraction中間打斷交易。玩家36格fingerprint、Core topology revision/conflict改變也要讓preview即時失效重算。
- **連線檢查不可載入 chunk**:Terminal/Import Bus/Export Bus 可 cache `access → Core` path，但每次使用前都要確認 path 仍連續、每格 chunk 已載入且仍是 network block；失效時只可用 `findLoadedNetworkPath` 尋找另一條已載入路徑。不可用 `getBlockEntity` 或一般 `getChunk` 為連線驗證偷載區塊，也不可只驗 path 兩端。
- **ItemKey 必須是真正 immutable snapshot**:`ItemKey.of` 要複製目前 `DataComponentMap`，不可直接保留來源 `ItemStack.getComponents()` 的 mutable patch view；來源 stack 後續 mutation 不得改變 map key 的 equals/hashCode。
- **燃料轉換是 server-authoritative Fuel-page action**:玩家選項只有 `Auto`/Fuel/Brew Energy；內部 `furnace_fuel` ID 不遷移。Fuel 不使用 item whitelist：slot 只做早期 UX，Core commit 重新呼叫目前 stack 的 `getBurnTime(null)`，正值才接受並精確按 burn ticks 加值，所以 logs/modded/data-map fuel 自然支援。`Auto` 首選 overflow 不得 fallback；失敗、換頁或關閉不得吞物。Installed machines 不因換頁/close 歸還。max-one instant station只接受一個並把remainder留在來源stack；Axe Energy只處理本次server驗證的transient input。本次repository格式不讀舊machine slot或legacy axe payload。
- **Bottle Energy 已退役且不遷移**:`BOTTLE_FUEL`/FuelTable mappings/lang/GUI target/Patchouli/test-world baseline與Core migration surface全部移除；現行repository schema不讀`bottle_fuel`或`legacyBottleFuel`。Crafting menu只保留retired 4-word zero data region維持100 data-slot client/server parity。
- **Cooking recipe cost 取 concrete recipe time**:exact cooking adapter 以每次從目前 server `RecipeManager` 重解析並重新 classify 的 holder `getCookingTime()`，同時計算 matching process pool 與 Fuel；preview/execute/EMI request 共用該 immutable match cost，非正值/stale/unsupported 不可補預設值。
- **Craftable mode 的候選不是 storage outputs**:`CraftableRecipeCatalog` 從目前 server recipes 與可用材料索引 candidates，再以目前材料/能量完整 simulate 後顯示；zero-storage Charcoal 可出現。Synthetic output 可選 exact recipe，但所有 extraction/Shift/pick/drop 路徑都 no-op。
- **Client crafting screen 不做 server 邏輯**:`CraftingTerminalScreen` 不掃 `RecipeManager`、不直接讀 core storage/energy；只讀 menu 重建的 immutable `RecipePresentation`。Exact recipe ID/kind、shaped dimensions/positions、shapeless state、output components/count、station、item/process/Fuel/tool available-required rows都由 server data/hidden slots 同步；tool resource 額外同步 explicit infinity bit，畫面只有該 bit 為 true 才顯示 `∞`，有限 `Long.MAX_VALUE` 仍是有限數。Selection clear 時保留 selected output identity，才能區分「尚未選擇」與中性 `No supported recipe`；empty presentation必須把diagram+ledger union當成單一中性prompt card。有效/空白recipe body都必須位於footer上方；ledger required resources由頂端對齊、最多四欄，超過八種時使用第三列，目前wire上限可容納總計十二種item/energy/tool resources；不可讓fullscreen高度變成大型無語意空區。有效 presentation 的 station badge 只由 shared screen 畫一次、固定在 diagram 右下並加暗色 overlay，native/EMI renderer不得重複畫；native slots使用vanilla inset slot frame，navigation與craft actions使用vanilla-style buttons。Craft Output item icon 必須依 server-synced destination 即時切換 player head/Storage Core。Wire regression 會把真實 signed-short data 與全部 hidden stacks複製到 client constructor 後再驗 exact model。Screen 只連 Magic Storage `RecipeDiagramRenderer`；dev compile/runtime基準是EMI `1.1.24+1.21.1`，release client dependency range是`[1.1.24,2)`且dedicated server不要求EMI。`ModList` guard 後的 compat adapter 只使用公開 `EmiApi`/`EmiRecipe`/`Widget`/`WidgetHolder`，exact ID/backing holder 必須一致，widget bounds 必須位於 holder/diagram；沒有 public representation才用native，runtime exception必須直接暴露，不可broad catch後靜默native。Ingredient predicate 只有 `equals` 相同才可合併；顯示 item 相同但 predicate 不同仍分列。SelfTest/GameTest 在 dedicated server 跑,client-only 類別仍不可被 server test 直接引用;Python static regressions會守護這點。Runtime `en_us`/`zh_tw` lang key set 必須完全相同。
- **合成 recipe/station contract / reload**:終端只支援 exact concrete `ShapedRecipe`、`ShapelessRecipe`、`StonecutterRecipe`、`SmeltingRecipe`、`BlastingRecipe`、`SmokingRecipe`、`CampfireCookingRecipe`、`SmithingTransformRecipe`、`SmithingTrimRecipe`，加上 internal synthetic axe strip/scrape/wax-off。這十個 built-in adapters 是目前單層合成唯一 family-policy authority；每個 `RecipeAdapterMatch` 都必填 ordered input predicates/multiplicity、stable `ResourceLocation` station descriptor、process/Fuel/tool cost、checked exact output/remainders、presentation semantics、source-backed variant resolver，以及 simulation/commit validators，不存在 default/generic recipe fallback。`CraftingTerminalMenu` 只保留通用 joint reservation、destination delivery、capacity planning 與 atomic mutation/rollback，不再保存 supported type、station或energy family table。Smithing template/base/addition直接委派原recipe predicate並綁exact來源stack；Transform與Trim都呼叫exact `assemble`，Trim另要求registry可解析material/pattern。同ID多variant的select/preview/Max/execute都從目前server `RecipeManager`重取holder並以selected exact output重解；計畫與commit之間holder identity不同、variant消失、missing station/context或cap超限即fail closed。Cooking adapter以目前holder的concrete cooking time同時計算matching process pool與Fuel；axe adapter每craft要求一單位finite/∞ tool cost，只有menu可實際扣除Core state。Axe catalog只允許`minecraft` namespace block，再掃representative vanilla axe + default state + `getToolModifiedState(..., simulate=true)`，不可用fabricated player/context呼叫任意mod block hook；只有Core finite/∞ Axe Energy可用時才建立synthetic candidates。Blocked output/ingredient failure不扣energy/tool，finite每craft扣一，infinite不扣。不同`Ingredient` predicate即使display item相同也不可按顯示signature合併；`use player inventory`只掃36格主背包+快捷列，不含armor/offhand。Max先算long-safe資源上限；若該批無法完整交付，再找destination可接收的最大完整craft數。Binary-search midpoint先提升為`long`；Core reservation超過`Integer.MAX_VALUE`時simulate/execute/rollback使用一次per-key long-count mutation並核對完整回傳值，不得逐int-safe chunk迴圈。
- **Direct output destination 與 EMI destination 必須分離**:`TerminalOutputDestination` 只控制 terminal 自己的 `×1/×8/×64/Max`，預設 Player，且與 `use player inventory` ingredient source 是兩個獨立 server-synced state。Player 在扣完 player-source ingredients 後先填玩家 36 格、overflow 回 Core；Storage 的 primary outputs 與 crafting remainders 全部只進 Core。完整批次的 type/long capacity 必須先驗證，少一件容量也整筆 no-op；commit 中途拒收則 rollback output 與 Core ingredients。EMI `CraftingDestination.NONE/CURSOR/INVENTORY` 仍是每個 exact request 的權威，不得被 terminal toggle 覆寫。
- **合成 output transaction 不得掉落/吞掉**:INVENTORY destination 先在扣除本次 player ingredients 的 36 格 snapshot 中放 primary output + container remainders，剩餘全規劃回 Core；若 post-extraction Core type/counter capacity 不能完整接受，就在任何 mutation 前拒絕整批。Commit 必須重新驗 player/Core/station/Axe Energy identity，再依序抽 Core ingredient、插 Core overflow、扣 process/Fuel/Axe Energy、套用 player inventory/cursor；Core mismatch 要顯式 rollback。Requested output 沒有 world-drop fallback，兩邊都滿就是 no-op。CURSOR 只接受單一 exact identity 且不可覆蓋 incompatible carried stack。
- **EMI dependency / compatibility contract**:EMI是release的client-only required dependency，Maven range固定`[1.1.24,2)`；`emi_version=1.1.24+1.21.1`只代表reproducible minimum compile/dev runtime，不得寫成玩家端exact pin。Full EMI只能放在既有client/data隔離source set的`fusionRuntimeRuntimeOnly`；plain `runtimeOnly`會污染dedicated GameTest，禁止使用。`scripts/resolve_emi_version.py`只接受同Minecraft 1.21.1、在inclusive-min/exclusive-max範圍內的numeric release；CI與release都要編譯minimum及最新相容API，client-smoke要放入最新相容full jar。Full jar一律由`./gradlew stageEmiRuntime -Pemi_version=...`走Gradle/TerraformersMC Maven resolver並先落在`build/client-smoke-mods`，不得直接寫共用`run/mods`；不要改回direct curl，本機已重現HTTP/2只傳部分jar後以curl 92中斷。解析/下載/ZIP驗證/編譯任一步失敗都直接fail，不得退回baseline假裝latest通過。Integration只能import`dev.emi.emi.api`公開契約；dedicated server不要求EMI。
- **EMI one-level magic crafting 綁 exact backing identity**:client gate先以backing recipe ID解析目前client holder，要求exact holder identity一致並重新classify；packet再送exact `{containerId,recipeId,amount,destination}`。NONE只選取，CURSOR/INVENTORY由server再次解析目前holder並由adapter重算station/Axe Energy/材料/實際cooking cost，並以「假想扣除本次 player ingredients 後」的 destination capacity 選不超過 request 的最大完整 recipe 數，再 atomic commit。不得信任 client craftability、不得換成同輸出第一個 recipe；Trim必須再綁selected exact output variant，special/dynamic/incomplete或缺少registry context仍fail closed。這是單層立即合成，不包含 RS2 recursive pattern tree，也不為 EMI synthetic favorites 同步整個 network inventory。
- **EMI recipe tree 不是server planner API**:目前最低支援EMI 1.1.24公開API可查input/output recipes、判斷`supportsRecipeTree()`與開啟tree screen，但建立/讀取goal使用runtime-only internal `BoM`/`MaterialTree`，API jar沒有這些類別。禁止直接link、reflection或把client tree當authority。多步驟合成依`docs/superpowers/plans/2026-07-14-modded-recipes-multistep-crafting.md`建立server adapter graph；EMI只作client recipe/tree UI。

## Decisions

- **統一能量池模型**:所有能量池同為 `Map<EnergyType, Long>` 無上限,差異只在來源,降低特例。
- **參考 RS2 而非依賴**:借設計模式,不加為依賴;加任何依賴前先問使用者(minimize deps)。
- **絕不整段照抄 RS2**:授權不同;只取模式自行實作。
- **一網一 core(multi-core 不支援)**:`rebuildNetwork` BFS 碰到第二個 core 即設 `conflicted` → 該 core 停止接受物品 + log 警告,避免雙重容量/非決定性;`isConflicted()` 可查、`tryIncrementalAdd` 在 conflicted 時退回 full rebuild。
- **Bus 吞吐 vs 省資源**:Import/Export bus 每個 cooldown(10 tick)搬「一整疊(≤64)」而非 1 個(吞吐 ×64、操作頻率不變 = 省資源);Export 只抽出目標放得下的量;Import active tick與passive capability共用missing-Core 10-tick negative cache，外部 automation 每次 probe不可重跑最高8192-block BFS。simulate 後 actual capacity 若因 reentrant handler 改變，必須以 commit 後 `ItemStack` remainder 為準精確退回；來源或 Core 拒收 rollback 時，把仍未處理的 remainder 顯式掉在 bus，而不是忽略回傳值吞物。
- **Bus cached core 必須驗證仍連通**:Import/Export bus 的 `cachedCore` 只有在 `core.getConnectedBlocks().contains(busPos)` 時可重用;中間 unit 被拆後不能靠舊 reference 繼續搬物。

## Hardening Pass (2026-06-21)

地毯式 bug 掃描 + 修正。關鍵點與借鏡的 RS2/AE2 模式(僅取模式,未照抄):

- **持久化**:`StorageCoreBlockEntity`只在storage reference改變時把chunk標記dirty；inventory、energy、machine與descriptor mutation直接更新attached `CoreStorageRecord`並讓overworld `CoreStorageRepository` dirty，outer mutation batch只mark一次。BE NBT固定只含`storageId`/`storageSchema`；完整`ItemStack` components與positive `long` count在world data中以每段最多63種的`inventorySegments`保存。拆除只新增recovery UUID→同一storage UUID的indirection，packed item沒有`BLOCK_ENTITY_DATA`或payload fallback。empty attached Core完成移除才刪record；non-empty、missing、corrupt、packed或duplicate狀態絕不建立空白replacement。
- **Core 身分 / remote 綁定**:repository record持有永久storage UUID與network UUID，BE只持有storage UUID/schema reference；packed token只指向同一record。Storage Core不要求特定工具才掉落。Remote Terminal綁`{network UUID,dimension,pos}`並只接受該位置仍解析到同一可用Core；舊coordinate-only binding必須fail closed要求重綁，不可因同座標新Core而靜默存取另一網路。`RemoteTerminalItem.isBound`必須驗齊x/y/z/dimension/UUID並用`tryParse`；重新綁定用`CustomData.update`只覆寫自己的keys。
- **匯流排/合成原子性**:Import bus active pull、passive capability insert 與 `craftItem` 都先 simulate/reserve 再 commit。Passive caller stack 不可 mutation，execute 只 commit simulate 可接受的數量並回 exact remainder。相同 predicate 也保留獨立需求；不同但重疊的 predicates(例如 `#planks` + oak-only，或 strict/partial component predicates)用 joint reservation/max-flow,同一個 item unit 不可算兩次,且要能把 broad match 重新路由給其他變體。conflicted Core、unsupported/dynamic/incomplete recipe 一律 preview 0/拒絕 execute;材料 commit 失敗要 rollback。物品級 crafting remainder(如 milk bucket→bucket)交還玩家。
- **insert/extract 契約**:`insertItem(stack, simulate)` 回傳實際接受量、`extractItem` 回傳實際取出量且庫存只減該量;呼叫端必須尊重回傳值(勿丟棄剩餘 → 否則網路滿時吃物)。
- **數值安全**:`consumeEnergy` / fuel 計算用 exact arithmetic；machine-generated energy 在 `Long.MAX_VALUE` 飽和，load 時負 energy 夾 0。item insert、NBT duplicate merge、fuzzy count 與 Core+玩家 preview 都必須 saturation，不得 wrap。
- **GUI**:隱藏的 `GhostSlot` 以 `isActive()` 停用。Vanilla 在 callback 後才設定 button focus，所以 callback 內清 focus 無效；Crafting screen 必須在 `super.mouseClicked(...)` 返回後清 `AbstractButton` focus並保留 search `EditBox`。Fuel 常駐 panel 不顯示 rate 公式或每格名稱，centered values 必須 `drawString(..., false)` 無 shadow，精確 identity/rate/total 放 tooltip；tooltip只在實際 station slot或reserve icon bounds觸發。Fuel target popup只用 `addWidget` 接收 input，再於 `super.render` 後的 foreground pass 繪製；覆蓋區必須先攔截click/scroll/tooltip，不能被 container foreground 蓋住或穿透到container slot。因 Slot.x/y final，client 重建 fuel/machine slots時要委派原 slot validation；off-page machine slots 放 `-9999,-9999`，panel-local wheel 切頁後重建。
- **網路**:所有 network block 都以 block callback 為 authoritative path：`onRemove` 在 state 已替換後重建 old position，`moved=true` 也不可跳過；`onPlace` 將一般、指令/`setBlock`、活塞 final state 合併到 next-tick growth pass，批次放置同 component 只重建受影響 Core。不可只靠 `BreakEvent`/`EntityPlaceEvent`。`tryIncrementalAdd` 先用 cached connected set 守住與 full rebuild 相同的 depth/member boundary；bridge detached segment、多 Core、破壞仍 full rebuild。Chunk lifecycle 另外以 server-only FULL ticket transition 驅動：失去 FULL 時只先保留既有 `LevelChunk` snapshot，下一 tick 才從 snapshot 的 network boundary 與仍 loaded 的外側 bounded BFS 找 Core 並 full rebuild；`ChunkEvent.Load` 只接受 server `LevelChunk` 並 one-shot 排程，所有 level interaction 延到下一 tick，flush 以 non-loading `getChunkNow` 試一次後一律移除 queue entry，未就緒不逐 tick retry，後續 FULL ticket transition 再補排；同一 flush window 內未實際卸載就重新取得 FULL 時取消 pending unload，不額外重掃。Boundary 會先篩出兩側都仍是 loaded network block 的有效連接，再套用 `MAX_NETWORK_BLOCKS` 上限；絕不 force-load、也不做每 tick/週期性整網掃描。Unload 會自動移除遠端容量，reload 會自動恢復，multi-Core conflict 與完整拓樸語意不變。`bfsFindCore` / `bfsFindCorePositions` / full `rebuildNetwork` 都受 `NETWORK_SCAN_DEPTH` + `MAX_NETWORK_BLOCKS` 上限保護;封包 handler 包 `ctx.enqueueWork(...)` 並透過 menu 的 exact-Core resolver。

## RS2 Heuristics Adopted (2026-06-21)

完整對照見 `docs/rs2-design-gap.md`(該採用 vs 故意分歧)+ 計劃 `docs/superpowers/plans/2026-06-21-rs2-heuristics-adoption.md`。已採用(僅取模式):

- **fuzzy 配對(A5b)**:合成材料以 `Ingredient.test` 跨變體;preview/execute 共用 joint reservation plan,最後按實際 `ItemKey` 精確扣除(取代各 predicate 獨立 `countMatching` 後重複算同一批物品)。
- **Action/Actor + change events(A2/A3 foundation)**:storage mutation 都可帶 `Action` 與 `Actor`;execute insert/extract 會發 `StorageListener` delta event。P3 若要做 grid delta,應基於此 listener,不要再從 client 保存 storage state。
- **view 設定 server 同步(A5a)**:sort / order / search 模式為 server 權威、經 data slot 同步;終端按鈕標籤/tooltip 反映真實狀態(非 client 私有)。
- **選取物品身分化(A4)**:crafting 選取以 `ItemKey` 身分為準(經畫面外 `selectionContainer` slot 自動同步),非 grid slot index → 排序/捲動後配方面板不再亂指。
- **合成 preview(A7)**:用 simulate 路徑算「可做 N / 缺什麼」顯示,不異動狀態。
- **增量網路成長(A1,安全範圍)**:放置 callback 先 coalesce 到 next tick；單一方塊接到既有 connected set 時，`StorageCoreBlockEntity.tryIncrementalAdd` 用 cached-set bounded path check 確認不越過 full rebuild depth/member guard 後加入(不讀整個 world graph)。**破壞/批次/不確定拓樸一律 full `rebuildNetwork`**(移除可能分裂網路,放置若 bridge detached segment 或連到多個 core 也屬不確定)。`capacityOf()` 是兩條路徑唯一的容量計算來源;GameTest 守護 simple growth、programmatic placement、piston move、depth boundary、detached bridge full rebuild、multi-core bridge conflicted。
- **仍未採用(未來)**:**P3 增量 grid delta**(grid 改收差異、不整份重建)與完整常駐 network graph;目前分頁式 grid(≤81 格,vanilla 已增量同步)低價值,除非改成整列表 client grid 才值得。
