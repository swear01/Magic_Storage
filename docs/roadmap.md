# Roadmap

## Backlog

- 擴充方塊(第二階段,PLAN §四)。
- Tier 3 遠端存取完整體驗(背包開關、遠端 GUI 優化)。
- Core item 大型 NBT redesign:極多 component-heavy variants 可能撞 vanilla network NBT 上限;需 server-side UUID/SavedData ownership 設計,不可截斷庫存。
- Recipe scope expansion：exact component-matched/preserving `SmithingTransformRecipe` 與 deterministic vanilla default-state axe strip/scrape/wax-off 已支援；mod block tool hooks、Smithing Trim、special/dynamic subclasses、event/player/world-state-dependent transforms、loom/cartography/grindstone/anvil/enchanting 仍需逐類建立可完整 simulate/commit 的 server contract，否則繼續 fail closed。
- Brewing production path:`BOTTLE_FUEL` / `BREW_ENERGY` 已有資料模型,但尚未接可執行的 brewing recipe consumer。
- 跨 mod machine/energy descriptor API：0.1.15 的兩列 Installed Stations `FlowGrid`、panel paging、右下 `fuelControlPanel` 與 selector rectangle 已不綁目前 descriptor 數量，但 `MachineEnergyTable`/energy pools 仍是 code-defined。未來要用 server-owned registry + client descriptor sync + 固定 menu slot/data parity/migration 契約支援第三方機器；target 種類過多時把現有 selector rectangle 升級成可捲動選單，不可用 client-local registration、動態 rail buttons 或會依 mod list 改變 container counts 的 naive API。
- Craft planner 完整性 hardening：目前 deterministic allocation/packing 已安全 fail closed，但後續要把 ingredient source 與 destination 聯合規劃，避免「改吃玩家材料即可騰出格子」仍被拒絕；primary output 與 container remainder 也要共同 packing，讓既有 Core stack/new inventory type 的可行交換不被固定順序漏掉。
- Component-sensitive craftable identity：Smithing catalog/display/selection 後續要保留每個 component-dependent output variant，並為無法由 registry-default display stack 建索引的 recipe 提供完整 fallback；deterministic source ordering 也要改用 canonical component identity，不依 `DataComponentMap.toString()`。
- NeoForge 一般 deprecation migration(低優先):`ClientSetup` for-removal bus 用法已清除；剩餘 `Level.hasChunkAt`、item registry holder 與 GameTest `RegistryFriendlyByteBuf` 舊 API 要在不改 unloaded-chunk 行為下遷移。
- Modrinth / CurseForge 發布:未來等公開頁面 metadata、icon、description、dependency matrix、secrets 與授權策略確認後再接進 release workflow;目前 CD 僅建立 GitHub Release。
